# 検索条件の永続化 仕様書

> 本書はリポジトリ検索における **qualifier とソートの端末保存・次回起動時の自動復元** に関する仕様を定義する。`requirements.md` および `docs/requirements/repository-search.md` を拡張する位置付けであり、本書と矛盾する場合は本書を優先する。プロジェクト全体方針（`@Observable` 採用、`NavigationStack`、Swift 6 strict concurrency など）は `requirements.md` および `CLAUDE.md` を参照し、本書では再掲しない。

## 1. 目的と背景

- リポジトリ検索を繰り返し使う利用者は、毎回同じ qualifier（言語・スター数など）とソートを手で設定し直しており、毎回の操作コストが高い。
- 本機能は、ユーザーが直近で適用した qualifier とソートを端末に保存し、次回アプリ起動時にデフォルトとして復元することで、同じ探索を最小操作で再現できるようにする。
- 想定ユーザーは、特定の言語・スター数レンジ・トピック・更新時期などで日常的にリポジトリを探索する開発者。
- 既存 PRD（`repository-search.md`）では §2.2 で「検索条件の永続保存」を将来課題（§11 の 2 番目）として切り出していた。本書はその将来課題のうち、**名前付きプリセットを伴わない単一の「直近条件」の永続化** を先行して具体化する。

## 2. スコープ

### 2.1 対象

- 適用済み qualifier（`in:` / `language:` / `stars:` / `pushed:` / `topic:`）の永続化
- ソート設定（ソートキー + 並び順）の永続化
- 次回アプリ起動時の qualifier・ソートの自動復元
- 保存条件のクリア挙動（全 qualifier 解除 + ソート既定復帰時に自動削除）
- 復元時のチップ列・ソート切替 UI の初期表示

### 2.2 対象外

以下は本フェーズでは扱わない。§9 に将来課題として再掲する。

- **キーワードの永続化**（次回起動時のキーワードは常に空。idle 状態から開始する）
- 検索結果リスト本体・スクロール位置・ページ番号の永続化（次回起動時に通信なしで前回結果を表示する機能は提供しない）
- 名前付きプリセット（複数の保存条件を切り替えて使う UI）
- 検索履歴
- 入力サジェスト
- iCloud / 端末間同期（本機能は端末ローカル保存のみ）
- 保存条件のエクスポート / インポート
- 「この条件を保存する」明示ボタン（保存は適用時に自動）
- Settings 画面からのクリア導線
- 多言語対応（i18n）
- 認証ユーザーごとの条件分離

## 3. ユーザーストーリーと受け入れ条件

### US-1: 直近の qualifier とソートが次回起動時に復元される

開発者として、前回設定した絞り込み条件とソートを、アプリを再起動しても引き継いで使いたい。

- AC-1.1 ユーザーが qualifier を「適用」した時点、または ソートキー / 並び順を変更した時点で、**そのときの qualifier 全体とソート全体** が端末に保存される。保存はユーザーから見えない（保存完了の通知 UI は出さない）。
- AC-1.2 ユーザーがチップ × による条件削除を行った時点で、**削除後の qualifier 全体** が端末に保存される。
- AC-1.3 アプリを完全終了して再起動した直後に検索画面を開くと、**保存されている qualifier がチップ列に復元** され、**保存されているソートがソート切替 UI の現在値として復元** される。
- AC-1.4 復元後の検索画面はキーワード未入力のため **idle 状態** で表示される。API は呼ばれない。idle 状態の案内メッセージは既存 PRD AC-1.1 と同じ「キーワードを入力してください」相当を表示する。
- AC-1.5 復元された状態で 1 文字以上キーワードを入力すると、復元済み qualifier とソートを反映したクエリで通常通り検索が発火する（既存 PRD AC-1.2 と同じ debounce 挙動）。
- AC-1.6 保存データが存在しない初回起動時は、qualifier が空、ソートが既定値（stars / desc）の状態で検索画面が表示される。idle 状態は既存 PRD AC-1.1 と一致する。

### US-2: 保存条件は明示操作なしに最新化される

開発者として、最後に使った条件が常に次回起動時のデフォルトであってほしい。古い条件が残らないでほしい。

- AC-2.1 qualifier の「適用」が行われた時点で、保存されている qualifier は **最新値で上書き** される（差分マージはしない）。
- AC-2.2 ソート変更が行われた時点で、保存されているソートは **最新値で上書き** される。
- AC-2.3 「保存する」「保存しない」を選ぶ確認 UI は出さない。保存は常に自動で行われる。
- AC-2.4 キーワード入力・キーワード変更・キーワード全消去・キーワードに連動する debounce / cancel・ページング・結果リストの更新は、**保存処理を一切引き起こさない**。

### US-3: 全条件を外せば保存データもクリアされる

開発者として、絞り込みを全部外して「真っ新な検索」に戻したいときは、次回起動時もその状態であってほしい。

- AC-3.1 適用済み qualifier がすべて空（`in:` / `language:` / `stars:` / `pushed:` / `topic:` のいずれも未設定）であり、かつソートが既定値（stars / desc）と一致する状態に到達した時点で、**保存されている qualifier・ソートは端末から削除** される。
- AC-3.2 削除はユーザーから見えない（削除完了の通知 UI は出さない）。
- AC-3.3 削除後にアプリを再起動した場合の挙動は AC-1.6（初回起動と同等）と一致する。
- AC-3.4 全 qualifier 解除をチップ × の連続削除で達成した場合も、編集 UI 内の「すべてクリア → 適用」で達成した場合も、結果は同じ（保存データの削除）。

### US-4: 保存データが破損していても安全に立ち上がる

開発者として、保存データが壊れていてもアプリが起動できなくなったりはしてほしくない。

- AC-4.1 保存されているデータがデコード失敗した場合、**保存データを無視して初回起動と同等の状態** で検索画面を表示する（qualifier 空 / ソート既定 / idle）。エラー表示は出さない。
- AC-4.2 デコード失敗時は、破損した保存データを端末から削除する（次回起動時に再度デコード試行で同じ失敗を繰り返さない）。
- AC-4.3 保存データに、現在のアプリで未知の値（例: 廃止された言語コード、未知のソートキー）が含まれていた場合、そのフィールドのみ既定値で補い、他のフィールドは復元する。フィールド単位の縮退ができないときは AC-4.1 と同じ扱いとする。

## 4. 振る舞い仕様

### 4.1 入出力一覧

| ユーザー操作 / 入力 | 前提条件 | 保存先への作用 | 画面への作用 |
|---|---|---|---|
| 初回起動でアプリを開く | 保存データなし | なし | qualifier 空、ソート既定 (stars / desc)、idle 状態 |
| 起動でアプリを開く | 保存データあり（正常） | なし | 保存値で qualifier・ソートを復元、キーワード空、idle 状態 |
| 起動でアプリを開く | 保存データあり（破損） | 破損データを削除 | qualifier 空、ソート既定、idle 状態（エラー表示なし） |
| qualifier 編集 UI で「適用」 | 適用後の qualifier が「空 + ソート既定」でない | qualifier を最新値で上書き保存 / ソートは現状値も含めて保存 | 既存 PRD AC-2.4 通り（チップ更新 + 再検索） |
| qualifier 編集 UI で「適用」 | 適用後の qualifier が空、かつソートが既定 | 保存データを削除 | 既存 PRD AC-2.4 通り |
| チップ × で 1 件削除 | 削除後の qualifier が空でない、またはソートが既定でない | qualifier を最新値で上書き保存 | 既存 PRD AC-2.8 通り |
| 最後のチップ × で qualifier を空に | 削除後にソートが既定 | 保存データを削除 | 既存 PRD AC-2.9 通り |
| ソートキー / 並び順を変更 | 変更後の値が既定でない、または qualifier が非空 | ソートを最新値で上書き保存（qualifier も併せて保存） | 既存 PRD AC-3.3 通り |
| ソートキー / 並び順を変更 | 変更後にソートが既定、かつ qualifier が空 | 保存データを削除 | 既存 PRD AC-3.4 通り |
| キーワード入力・変更・全消去 | — | 何も保存しない / 削除もしない | 既存 PRD §3 US-1 通り |
| ページング・再試行・結果行タップ・画面離脱 | — | 何も保存しない / 削除もしない | 既存 PRD 通り |

### 4.2 状態と表示

本機能は既存 PRD `repository-search.md` §4.2 の 8 状態（idle / loading / loaded / no-results / error-network / error-rate-limited / paging-loading / paging-error）を **変更しない**。追加の状態も導入しない。

復元の影響を受けるのは「画面表示時点の初期値」のみ。

| 観点 | 既存 PRD の初期値 | 本 PRD による上書き |
|---|---|---|
| qualifier 初期値 | `RepositorySearchQualifiers.empty` 相当（空） | 保存データがあれば保存値、なければ空 |
| ソート初期値 | stars / desc | 保存データがあれば保存値、なければ stars / desc |
| キーワード初期値 | 空 | 変更なし（常に空） |
| 検索画面オープン直後の phase | idle | 変更なし（idle のまま。キーワード空のため API 発火条件を満たさない） |

### 4.3 バリデーションと異常系

#### 4.3.1 保存処理の異常

- 保存中の I/O 失敗（端末ストレージ書き込みエラー等）は **ユーザーに表示しない**。次回起動時の復元値が古いままになるだけで、現在のセッションの検索動作には影響しない。
- 保存処理は同期的に完了する想定（小さな JSON を UserDefaults に書く想定）。

#### 4.3.2 復元時の破損データ

- §3 AC-4.1〜AC-4.3 に従う。
- 破損判定は「JSON デコードの失敗」「必須フィールドの欠落」「フィールド単位の縮退も不可能な未知値」。
- いずれの場合も、復元処理がアプリ起動を阻害しない。

#### 4.3.3 復元後の通常異常系

- 復元後に発火する検索の異常系は、既存 PRD §4.3 の network / rate-limited / cancel 3 分類にそのまま従う。本書では追加しない。

## 5. データ要件

### 5.1 保存対象

以下のフィールドを 1 つのまとまり（単一の保存スロット）として保存する。

- qualifier 全体（`in:` の選択集合、`language:`、`stars:` の最小・最大、`pushed:` の開始・終了、`topic:` のリスト）
- ソート（ソートキー、並び順）

キーワード、結果リスト、ページ番号、スクロール位置、エラー状態、debounce タイマー状態は **保存対象外**。

### 5.2 保存先

- 端末ローカルの永続ストレージに 1 つのスロット（単一の保存値）として保存する。
- 同期や iCloud 連携は行わない。
- 名前付きプリセットを意図しないので、複数スロットは持たない。

### 5.3 保存値の同等性

- §3 US-3 における「既定状態」とは以下を全て満たす状態を指す。
  - qualifier が `RepositorySearchQualifiers.empty` と等価（`isEmpty == true`）
  - ソートが `RepositorySearchSort.default`（stars / desc）と等価
- これら 2 条件が同時に満たされたタイミングで、保存データを削除する。

### 5.4 復元時の縮退ルール

- 言語が「現在のアプリで未知」だった場合、`language:` のみ未設定として復元し、他の qualifier は保存値どおり復元する。
- ソートキーまたは並び順が未知だった場合、ソート全体を既定値（stars / desc）として復元し、qualifier は保存値どおり復元する。
- 上記の縮退で復元できない構造的破損は AC-4.1 に従い、全体を初期状態として扱う。

## 6. 画面構成

本機能では新規画面を追加しない。既存の検索画面（`RepositorySearchView`）の **初期値の出所が変わるだけ** とする。

- 検索バー: キーワードは常に空で起動する。
- チップ列: 保存値があれば復元時点で 1 件以上のチップが描画され、キーワードチップは出ない。
- ソート切替 UI: 保存値があれば復元時点でその値が「現在のソート」として描画される。
- 結果リスト: 起動直後は idle のため非表示。キーワードを入力した時点から既存 PRD 通りの挙動。

クリア専用 UI（Settings 画面、リセットボタン等）は本フェーズでは追加しない。クリアは AC-3.1 に従い、qualifier 全解除 + ソート既定復帰の操作の副作用として行われる。

## 7. 非機能要件

### 7.1 性能

- 検索画面の最初の描画から、復元済み qualifier とソートが反映された状態でユーザーに見えるまで、**150ms 以内** に完了すること（保存データのデコード + 状態反映の合計）。
- 保存処理（適用 / ソート変更 / チップ削除のたびに 1 回）は **1 操作あたり 50ms 以内** に完了すること。
- 保存処理はメインスレッドをブロックしないか、ブロックしても上記時間に収まること（小さな JSON を UserDefaults に書く想定で許容）。

### 7.2 アクセシビリティ

- 復元によって描画されたチップ・ソート切替の VoiceOver ラベルは、既存 PRD `repository-search.md` §8.2 と同等の文言・粒度であること。
- 復元自体に追加のアクセシビリティ通知（読み上げ）は出さない。

### 7.3 プライバシー

- 保存されるのは qualifier とソートのみ。キーワード・取得結果・閲覧履歴は端末に残らない。
- 保存場所は端末ローカルのみ。サーバー送信・他端末同期はしない。

## 8. 完了の定義

以下のすべてを満たしたとき、本機能は完了とみなす。

- §3 のすべての受け入れ条件（AC-1.1〜AC-4.3）が満たされる。
- §4.3 の異常系（保存 I/O 失敗、復元データ破損、復元時の縮退）が定義通りに機能する。
- §5 の保存対象・縮退ルールに沿って保存・復元が行われる。
- §7 の非機能目標を満たす。
- §10 の検証要件のテストが Swift Testing ですべて通る。
- 既存 PRD `repository-search.md` の AC-1.1〜AC-6.2 が **本 PRD 適用後も引き続き満たされる**（特に AC-3.2 の既定ソート、AC-2.4 の適用後再検索）。

## 9. 将来課題

§2.2 の対象外項目のうち、優先度の高いものから列挙する。

1. 名前付きプリセット（複数の保存条件を切り替える UI）
2. 検索履歴（過去 N 件のキーワード / 条件の一覧 + 再適用）
3. キーワードの永続化（直近キーワードの自動補完 / 再起動時の再入力）
4. 保存条件のクリア導線（Settings 画面の明示ボタン）
5. 結果リストとスクロール位置のオフライン復元
6. iCloud / 端末間同期
7. 入力サジェスト
8. 認証ユーザーごとの条件分離（複数アカウント切り替え時）

## 10. 検証要件と AC の対応

§3 の各受け入れ条件に対し、Swift Testing で検証すべき観点を定義する。具体的なテストコードや Mock 構造には踏み込まない。

| 検証観点 | 対応 AC |
|---|---|
| qualifier 適用時に保存スロットが最新値で上書きされる | AC-1.1, AC-2.1 |
| ソート変更時に保存スロットが最新値で上書きされる | AC-1.1, AC-2.2 |
| チップ × による削除時に保存スロットが最新値で上書きされる | AC-1.2 |
| 起動時に保存スロットがあれば qualifier・ソートが復元される | AC-1.3 |
| 復元後のキーワードは空、phase は idle、API 未発火 | AC-1.4, AC-2.4 |
| 復元後にキーワードを入力すると復元済み条件込みで検索発火 | AC-1.5 |
| 保存スロットなしの初回起動は qualifier 空・ソート既定・idle | AC-1.6 |
| 保存タイミングで確認 UI が出ない（保存は自動・無音） | AC-2.3 |
| キーワード操作・ページング・結果更新では保存・削除が発生しない | AC-2.4 |
| qualifier 空 + ソート既定に到達した時点で保存スロットが削除される | AC-3.1, AC-3.4 |
| 保存削除はユーザーに見える通知を発生させない | AC-3.2 |
| 削除後の再起動は初回起動と同じ状態 | AC-3.3 |
| デコード失敗時に保存データを無視して初回起動同等で起動 | AC-4.1 |
| デコード失敗時に破損データが端末から削除される | AC-4.2 |
| 未知の言語値 / ソート値はフィールド単位で既定に縮退して復元 | AC-4.3 |
| 復元 + 描画完了が 150ms 以内、保存処理が 50ms 以内（自動測定可能な範囲で） | §7.1 |
| 既存 PRD `repository-search.md` の AC が本機能適用後も通る | §8 |
