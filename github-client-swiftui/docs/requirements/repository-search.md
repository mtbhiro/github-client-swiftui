# リポジトリ検索 仕様書

> 本書はリポジトリ検索機能に関する仕様を定義する。`requirements.md` の「3.1 リポジトリ検索」を拡張する位置付けであり、本書と矛盾する場合は本書を優先する。プロジェクト全体方針（`@Observable` 採用、`NavigationStack` ベースのナビゲーション、Swift 6 strict concurrency、画像ローダーの差し替え可能性、debounce の枠組み等）は `requirements.md` および `CLAUDE.md` を参照し、本書では再掲しない。

## 1. 目的と背景

- モバイル GitHub クライアント利用者が、キーワードと qualifier（言語・スター数・最終 push 日・トピック・検索対象）で素早く絞り込み、stars / updated でソートしながら **単発の探索を高速に終わらせる** ことを目的とする。
- 想定ユーザーは、開発中に「特定のテーマで活発に更新されているリポジトリを探したい」「自分の関心言語に閉じて人気リポジトリを発掘したい」といった一過性の調査を行う iOS 利用者。
- 既存スコープではキーワード単独の検索しか提供されておらず、結果が膨大かつ関連度の低いリポジトリが上位に出る課題がある。本機能で qualifier とソートを導入し、検索結果の質と発見効率を向上させる。

## 2. スコープ

### 2.1 対象

- 検索画面（キーワード入力欄、適用中条件チップ列、ソート切替、結果リスト）
- 条件編集 UI（言語 / スター数 / 最終 push 日 / トピック / `in:`）
- GitHub Search API (`GET /search/repositories`) の `q` / `sort` / `order` / `per_page` / `page` パラメータ構築
- 検索結果リストの状態表示（idle / loading / loaded / no-results / error / paging）
- ページング（per_page=30、上限 1000 件、リスト末尾でのインクリメンタル読み込み）
- 結果行タップによるリポジトリ詳細画面への遷移

### 2.2 対象外

以下は本フェーズでは扱わない。§10 に将来課題として再掲する。

- アカウント連携・OAuth 認証
- 検索条件の永続保存・名前付きプリセット・検索履歴
- 入力サジェスト（過去入力からの補完、候補表示）
- ユーザー検索 / Issue 検索
- リポジトリのブックマーク・お気に入り
- オフライン表示・キャッシュ表示
- ソートキー `forks` / `help-wanted-issues` の提供
- 同梱言語リスト（後述 §5.3）に含まれない言語の検索（自由入力フォールバックは設けない）
- Pull to Refresh
- 多言語対応（i18n）。本フェーズの UI 文言は日本語単一とする
- OR / NOT による条件式の入れ子、先頭ハイフンによる除外
- カスタムプロパティ (`props.*`)
- 表示モード（List / Grid 等）の切り替え
- リポジトリ詳細以降の画面仕様（別 PRD で扱う）

## 3. ユーザーストーリーと受け入れ条件

### US-1: キーワードで素早く検索する

開発者として、興味のあるテーマを入力するだけで、関連リポジトリを即座に確認したい。

- AC-1.1 検索画面を開いた直後（未入力・未適用）は、結果リストの代わりに **idle 状態の案内**（「キーワードを入力してください」相当のメッセージ）が表示される。API は呼ばれない。
- AC-1.2 キーワード欄に 1 文字以上入力し、入力停止から 300ms 経過した時点で API リクエストが 1 回発火し、リクエスト中は **loading 表示**（プログレスインジケータ）が画面中央に表示される。
- AC-1.3 連続入力中は最後の入力から 300ms 経過するまで API は呼ばれない。途中で再入力された場合、進行中の処理はキャンセルされ、ユーザーにエラーは表示されない。
- AC-1.4 結果が 1 件以上返れば、リスト形式で各リポジトリの **owner avatar / `<owner>/<name>` / description / stars / language** が表示される。
- AC-1.5 結果が 0 件のとき、**no-results 状態** として「`"<入力キーワード>"` に一致するリポジトリは見つかりませんでした」というメッセージが表示される。キーワード部分は引用符で囲まれる。
- AC-1.6 キーワードを全消去した時点で、結果リストは即時クリアされ、idle 状態に戻る。API は呼ばれない。

### US-2: qualifier で絞り込む

開発者として、言語やスター数で絞ることで、ノイズの少ない結果一覧を得たい。

- AC-2.1 検索画面のフィルタ操作要素から条件編集 UI を開ける。
- AC-2.2 編集 UI で言語・スター数（最小・最大）・最終 push 日（以降 / 以前 / 範囲）・トピック（複数）・検索対象 `in:`（name / description / readme / topics の複数選択）を編集できる。
- AC-2.3 編集 UI 上の値変更は、**「適用」操作を行うまで API リクエストを発生させない**。
- AC-2.4 「適用」操作を行うと、編集 UI が閉じ、検索画面に戻り、検索バー直下に **適用中の条件をチップ列として** 表示する。同タイミングで再検索が発火する。
- AC-2.5 「キャンセル」操作で編集 UI を閉じると、編集中の値は破棄され、適用済み条件には影響しない。
- AC-2.6 「すべてクリア」操作で、編集中の全条件が初期状態に戻る。編集 UI は閉じない。
- AC-2.7 数値範囲が不正（最小 > 最大、または負の数）、または日付範囲が不正（開始 > 終了）の間は「適用」操作が無効化される。
- AC-2.8 適用済みの条件チップは × アイコンで個別に削除でき、削除した時点で再検索が発火する。
- AC-2.9 チップを最後の 1 件まで削除し、かつキーワードも空のとき、結果リストは idle 状態に戻る。

### US-3: ソートを切り替える

開発者として、人気順と更新順を切り替えて、自分の関心軸で結果を並び替えたい。

- AC-3.1 検索画面にソート切替操作要素があり、ソートキー（stars / updated）と並び順（asc / desc）を変更できる。
- AC-3.2 初期値はソートキー stars、並び順 desc（降順）である。
- AC-3.3 ソートキーまたは並び順のいずれかを変更した時点で、進行中の検索リクエストはキャンセルされ、新しいソートで再検索が発火する。再検索中は loading 表示に戻り、前の結果は即時クリアされる（並びだけ変わった結果が一瞬見える状態を発生させない）。
- AC-3.4 キーワードが空かつチップが 0 件の状態でソートだけを変更しても、API は呼ばれず idle 状態のままである。

### US-4: スクロールで続きを読み込む

開発者として、最初の 30 件で見つからなかった場合に、スクロールするだけで次の 30 件を見られるようにしたい。

- AC-4.1 結果リスト末尾までスクロールし、リスト末尾に配置された **追加読み込みインジケータが画面に表示された時点** で次ページが取得される。
- AC-4.2 取得中はリスト末尾に loading インジケータが表示され、既存の結果は保持される。多重リクエストは発生しない。
- AC-4.3 次ページ取得が成功すると、既存リストの末尾に新規アイテムが追加表示される。
- AC-4.4 累積取得件数が **1000 件に達したとき**、リスト末尾に「これ以上の結果は表示できません」相当の **終端メッセージ** が表示され、それ以上のページング要求は発生しない。
- AC-4.5 次ページ取得が失敗したとき、既存リストは保持されたまま、リスト末尾に **再試行ボタン** が表示される。再試行ボタンをタップすると同じページの取得が再実行される。
- AC-4.6 再検索が発火した場合、ページング状態（取得済みカーソル / 終端フラグ）はリセットされる。

### US-5: 異常系を理解できる形で表示する

開発者として、通信エラー時には何が起きたかと、どう回復するかを把握したい。

- AC-5.1 ネットワーク失敗（オフライン / タイムアウト / 5xx）が発生したとき、結果リストの代わりに **error-network 状態**（メッセージと再試行ボタン）が表示される。再試行ボタンをタップすると同じ条件で再検索が発火する。
- AC-5.2 GitHub API のレート制限超過（HTTP 403 + `X-RateLimit-Remaining: 0`、または HTTP 429）が発生したとき、**error-rate-limited 状態** が表示される。表示項目は次の 3 つに限定する。
  - 種別: 「API 利用制限に達しました」相当の固定文言
  - 解除絶対時刻: `X-RateLimit-Reset` から取得した時刻を端末ローカルタイムゾーンで `YYYY-MM-DD HH:mm` 形式で表示。カウントダウンや残り秒表示は行わない
  - 再試行手段: 再試行ボタン（即時タップ可能。タップ時にレートが回復していなければ再度同じ画面に戻る）
- AC-5.3 ユーザー操作起因のキャンセル（キーワード変更 / qualifier 適用 / チップ削除 / ソート切替 / 画面離脱）で発生する `CancellationError` は **エラーとして表示しない**。直前の状態（idle / loading / loaded / 各 error）は変化しない。
- AC-5.4 再検索（キーワード変更・適用・チップ削除・ソート切替）が発火した場合、進行中のエラー状態は破棄され、loading 状態に切り替わる。

### US-6: 結果から詳細へ進む

開発者として、興味を持ったリポジトリの詳細を確認したい。

- AC-6.1 結果リストの行をタップすると、**リポジトリ詳細画面に遷移** する。詳細画面の仕様は本書のスコープ外。
- AC-6.2 詳細画面から検索画面に戻ったとき、検索条件・ソート・取得済み結果・スクロール位置は **保持される**。

## 4. 振る舞い仕様

### 4.1 入出力一覧

| ユーザー操作 / 入力 | 前提条件 | システム応答 / 表示 |
|---|---|---|
| 検索画面を初回表示 | — | idle 状態（案内メッセージ）。API 未発火 |
| キーワードに 1 文字以上入力 | — | 最終入力から 300ms 後に API 発火、loading 表示。前結果は即時クリア |
| キーワード入力中に追加入力 | 前回入力から 300ms 未満 | 待機タイマーをリセット、API 未発火 |
| キーワードを全消去 | キーワードが空でない状態 | 結果クリア、idle 状態に復帰、進行中リクエストはキャンセル |
| 編集 UI で値を変更 | 編集 UI 表示中 | API 未発火。「適用」操作の有効/無効のみ更新 |
| 編集 UI で「適用」 | 入力値がすべて妥当 | 編集 UI が閉じ、チップ列が更新され、再検索発火、loading 表示。前結果は即時クリア |
| 編集 UI で「キャンセル」 | 編集 UI 表示中 | 編集中の値は破棄、適用済み条件は不変、API 未発火 |
| 編集 UI で「すべてクリア」 | 編集 UI 表示中 | 編集中値が初期状態にリセット、編集 UI は閉じない、API 未発火 |
| チップの × をタップ | 適用中条件が 1 件以上 | 該当条件を除去、チップ列を更新、再検索発火、前結果クリア |
| 最後のチップ × をタップ | チップ 1 件かつキーワード空 | チップ列が空に、idle 状態に復帰、API 未発火 |
| ソートキーまたは並び順を変更 | キーワードまたはチップが 1 件以上 | 進行中処理をキャンセル、再検索発火、前結果クリア |
| ソート変更（空クエリ） | キーワード空かつチップ 0 件 | API 未発火、idle 状態維持 |
| 結果末尾のローディングが画面に表示 | 累積件数 < 1000 かつ次ページあり | 次ページ API 発火、リスト末尾に loading 表示 |
| 結果末尾のローディングが画面に表示 | 累積件数 ≥ 1000 | リスト末尾に終端メッセージ表示、API 未発火 |
| 次ページ取得失敗 | ページング中 | 既存結果保持、リスト末尾に再試行ボタン表示 |
| 再試行ボタンタップ（ページング） | paging-error 状態 | 同ページを再取得、リスト末尾に loading 表示 |
| 再試行ボタンタップ（初回検索） | error-network / error-rate-limited 状態 | 同じ条件で再検索発火、loading 表示 |
| 結果行タップ | loaded 状態 | リポジトリ詳細画面へ遷移 |
| 詳細画面から戻る | — | 検索条件・ソート・取得済み結果・スクロール位置を保持して復元 |
| 連続適用（適用直後にもう一度編集 UI を開く） | 適用直後 | 編集 UI には現在の適用済み条件が初期値として表示される |
| 画面離脱（戻る・タブ切替等） | 進行中リクエストあり | リクエストはキャンセルされる、エラー表示は出さない |

### 4.2 状態と表示

検索画面が取りうる状態は以下の 8 つ。各状態でユーザーから観測できる表示と、次に遷移しうる状態を定義する。

| 状態 | ユーザーから見える表示 | 主な遷移先 |
|---|---|---|
| **idle** | 案内メッセージ。結果リストなし。リスト関連の操作要素は出さない | キーワード入力 / 適用 / ソート変更（クエリ非空のとき）→ loading |
| **loading** | 画面中央のプログレスインジケータ。前の結果は表示されない | 成功 → loaded / no-results、失敗 → error-network / error-rate-limited、入力変更 → loading（再開）、キーワード全消去 → idle |
| **loaded** | リポジトリのリスト。各行に owner avatar / `<owner>/<name>` / description / stars / language。末尾に次ページの状態（loading / 終端 / 再試行）を表示する領域 | 末尾到達 → paging-loading、再検索発火 → loading、行タップ → 詳細画面遷移 |
| **no-results** | `"<キーワード>"` を引用符で囲んだ「一致するリポジトリは見つかりませんでした」メッセージ | 再検索発火 → loading |
| **error-network** | エラーメッセージと再試行ボタン。リストなし | 再試行 → loading、再検索発火 → loading |
| **error-rate-limited** | 種別文言・解除絶対時刻 (`YYYY-MM-DD HH:mm`)・再試行ボタン。リストなし | 再試行 → loading、再検索発火 → loading |
| **paging-loading** | loaded のリスト + 末尾に loading インジケータ | 成功 → loaded（リスト末尾に追加）、失敗 → paging-error、累積 1000 達成 → loaded（末尾に終端メッセージ）、再検索発火 → loading |
| **paging-error** | loaded のリスト + 末尾に再試行ボタン | 再試行 → paging-loading、再検索発火 → loading |

idle と no-results は **別状態として扱う**（idle は「未検索」、no-results は「検索したが 0 件」）。

### 4.3 バリデーションと異常系

#### 4.3.1 入力バリデーション

- **スター数**: 整数のみ。最小値・最大値ともに 0 以上。最小 > 最大は不正。両方未指定は条件として付与しない。
- **最終 push 日**: ISO8601 (`YYYY-MM-DD`) のみ。範囲指定時、開始 > 終了は不正。未来日付は許容（API 結果に従う）。
- **トピック・キーワード**: 前後の空白はトリムし、トリム後に空文字となる入力は条件として無視する。
- **言語**: 同梱リスト 30 件からの選択のみ。それ以外は入力不能。

不正な値が含まれている間、編集 UI の「適用」操作は無効化される（AC-2.7）。

#### 4.3.2 異常系の分類と挙動

異常系は 3 分類で扱う。

| 分類 | 該当条件 | 挙動 |
|---|---|---|
| **network** | URLError 系（オフライン / タイムアウト / 名前解決失敗）、HTTP 5xx | error-network 状態に遷移。再試行ボタンで同条件再検索 |
| **rate-limited** | HTTP 403 + `X-RateLimit-Remaining: 0`、または HTTP 429 | error-rate-limited 状態に遷移。`X-RateLimit-Reset` の絶対時刻を表示 |
| **cancel** | `CancellationError`（ユーザー操作起因） | エラー表示しない。状態は変化させない |

#### 4.3.3 キャンセル要因（cancel 分類）

以下のいずれかが発生したとき、進行中の検索 / ページングリクエストは協調的にキャンセルされる。キャンセル結果は UI に反映しない（AC-5.3）。

- キーワードの変更（debounce 経由含む）
- 編集 UI での「適用」
- チップ × による条件削除
- ソートキー / 並び順の変更
- 画面離脱（タブ切替・ナビゲーションスタック pop 等）

#### 4.3.4 再検索時のエラー状態の扱い

- 再検索（キーワード変更 / 適用 / チップ削除 / ソート切替）が発火した時点で、現在の error-network / error-rate-limited 状態は破棄され、loading 状態に切り替わる（AC-5.4）。
- ページング失敗中（paging-error）に再検索が発火した場合も、既存結果は破棄され loading に戻る。

## 5. 検索条件仕様

GitHub Search API の `q` パラメータに対するシリアライズ規則を定義する。実装は本書のルールに対し Swift Testing で文字列比較可能でなければならない。

### 5.1 全体規則

- `q` は **キーワード** と **複数の qualifier** を空白区切りで連結した文字列とする。
- 同一 qualifier の複数指定は GitHub API の仕様に従う（API 側の AND / OR 解釈は API に委ね、本書では断定しない）。本書はあくまで「クライアントとして qualifier をどうシリアライズするか」までを規定する。
- 値に空白を含む場合は `"..."` で囲む。
- 不正値はシリアライズ対象から除外する。
- qualifier の並び順は **決定的** とする。並びは次の固定順とする（テストの安定化のため）。
  1. `in:`
  2. `language:`
  3. `stars:`
  4. `pushed:`
  5. `topic:`（複数ある場合は適用順を保つ）
- キーワードは qualifier 群より前に置く。

### 5.2 qualifier ごとのシリアライズ

| qualifier | 値の形 | シリアライズ結果 |
|---|---|---|
| キーワード | フリーテキスト 1 件 | テキストそのまま（空白を含む場合は `"..."` で囲む）|
| `in:` | name / description / readme / topics の複数選択 | **カンマ区切り単一 qualifier**: `in:name,description`（選択順は §5.1 の固定順に従い `name, description, readme, topics` の順で並べる） |
| `language:` | 同梱リストから単一選択 | `language:Swift` のように Linguist 英語表記をそのまま使用。値に空白を含む言語名（例: `Vim Script`）は `language:"Vim Script"` |
| `stars:` 最小のみ | 整数 N | `stars:>=N` |
| `stars:` 最大のみ | 整数 M | `stars:<=M` |
| `stars:` 範囲 | N, M（N ≤ M） | `stars:N..M` |
| `pushed:` 以降のみ | `YYYY-MM-DD` | `pushed:>=YYYY-MM-DD` |
| `pushed:` 以前のみ | `YYYY-MM-DD` | `pushed:<=YYYY-MM-DD` |
| `pushed:` 範囲 | 開始, 終了 | `pushed:YYYY-MM-DD..YYYY-MM-DD` |
| `topic:` | 文字列、複数指定可 | `topic:<value>` を空白区切りで複数並べる（例: `topic:ios topic:swiftui`）|

### 5.3 言語候補リスト

- 同梱する言語は **30 件**。GitHub Linguist の英語表記をそのまま採用する。日本語訳は行わない。
- 30 件の選定は「GitHub 上のリポジトリ数が多く、検索でよく使われる主要言語」を基準とする。具体的な 30 件のリストは実装着手時に確定するが、選定基準として次を満たすこと。
  - スクリプト系・コンパイル系・モバイル系・Web 系の主要言語を網羅する
  - 30 件外の言語（Linguist でカバーされる数百言語）はサポートしない（§2.2）
- 候補表示・選択 UI 上での言語名は **Linguist 英語表記そのまま**（例: `Swift`、`JavaScript`、`C++`、`Vim Script`）。

### 5.4 API パラメータ

| パラメータ | 値 |
|---|---|
| `q` | §5.1〜§5.2 で構築した文字列 |
| `sort` | `stars` または `updated`（必須、未指定は提供しない） |
| `order` | `asc` または `desc`（必須） |
| `per_page` | `30`（固定） |
| `page` | 初回 `1`、ページングで `+1` |

## 6. チップ表記規則

検索バー直下のチップ列に表示する文字列は以下の規則で構築する。Swift Testing で文字列等価比較できる粒度で固定する。最終的なフォントや余白は実装に委ねるが、**ラベル文字列は本規則に従う**。

| qualifier | チップラベルのテンプレート | 例 |
|---|---|---|
| キーワード | `"<keyword>"` | `"swiftui"` |
| `in:` 複数選択 | `in: <a>, <b>, ...`（カンマスペース区切り、§5.1 の固定順）| `in: name, description` |
| `language:` | `<Language>`（Linguist 英語表記そのまま、空白含みも `"..."` で囲まない平文）| `Swift`、`Vim Script` |
| `stars:` 最小のみ | `★ ≥ N` | `★ ≥ 100` |
| `stars:` 最大のみ | `★ ≤ M` | `★ ≤ 5000` |
| `stars:` 範囲 | `★ N–M`（en-dash `–` 区切り）| `★ 100–5000` |
| `pushed:` 以降のみ | `pushed: ≥ YYYY-MM-DD` | `pushed: ≥ 2025-01-01` |
| `pushed:` 以前のみ | `pushed: ≤ YYYY-MM-DD` | `pushed: ≤ 2025-12-31` |
| `pushed:` 範囲 | `pushed: YYYY-MM-DD – YYYY-MM-DD`（en-dash `–` 前後に半角スペース）| `pushed: 2025-01-01 – 2025-12-31` |
| `topic:` 複数 | `#<topic>` を **複数チップに分割** して表示 | トピック 2 件なら `#ios` と `#swiftui` の 2 つのチップ |

補足:

- キーワードは独立した 1 チップとして表示する。キーワードが空のときはチップを出さない。
- `in:` は複数選択でも 1 チップにまとめる（×タップで全 `in:` 選択をまとめて解除）。
- `topic:` は仕様上 1 トピックにつき 1 チップとする（× タップで個別解除）。
- チップの並びは §5.1 の qualifier 並び順に揃える。`topic:` が複数ある場合は適用順を保つ。

## 7. 異常系（再掲と固有挙動）

§4.3 を踏まえ、本機能で固有のポイントを補足する。

- **error-network**: メッセージは「通信に失敗しました」相当の固定文言 + 再試行ボタン。具体的な原因（オフライン / タイムアウト / 5xx）の細分表示は行わない。
- **error-rate-limited**: 表示する情報は「種別 / 解除絶対時刻 / 再試行手段」の 3 項目のみ。`X-RateLimit-Reset` ヘッダが取得できない異常系では、解除絶対時刻の代わりに「時刻不明」と表示する。
- **paging-error**: 既存結果は維持し、リスト末尾に再試行ボタンを置く。エラーをトースト等で別途出すことはしない。
- **cancel**: いかなる UI 表示も行わない。状態遷移を発生させない。

## 8. 非機能要件

### 8.1 性能

- API レスポンスの **p95 が 1.5 秒以内** に loaded / no-results / error 状態に遷移すること（API 応答到達からの UI 反映時間）。
- ローカル処理（クエリ文字列構築、チップラベル生成、状態遷移計算など）は **1 操作あたり 100ms 以内** に完了すること。
- ページング 1 回あたりの取得件数は 30 件、累積上限は 1000 件。

### 8.2 アクセシビリティ

- iOS 標準の Dynamic Type に追従し、文字サイズ拡大時もチップ列・リスト行の主要情報がトランケートのみで欠落しないこと。
- 各操作要素（検索バー、編集 UI 起動、チップ × 、ソート切替、再試行ボタン、結果行）に **アクセシビリティラベル** を付与し、VoiceOver で操作可能であること。

### 8.3 ローカライズ

- 本フェーズは UI 文言を日本語単一とする。
- 言語候補リスト（§5.3）および `language:` チップラベルは Linguist 英語表記そのまま。
- 多言語化は将来課題（§10）。

## 9. 完了の定義

以下のすべてを満たしたとき、本機能は完了とみなす。

- §3 のすべての受け入れ条件（AC-1.1〜AC-6.2）が満たされる。
- §4.3 の異常系 3 分類すべてに対し、定義された表示と回復手段が機能する。
- §5 のクエリ文字列構築規則がすべての qualifier の組み合わせで仕様通りに動作する。
- §6 のチップ表記規則がすべての qualifier に対し仕様通りに描画される。
- §10 の検証要件のテストが Swift Testing ですべて通る。
- §8 の非機能目標を満たす（性能 p95 / Dynamic Type / VoiceOver ラベル）。

## 10. 検証要件と AC の対応

§3 の各受け入れ条件に対し、Swift Testing で検証すべき観点を定義する。具体的なテストコードや Mock 構造には踏み込まない。

| 検証観点 | 対応 AC |
|---|---|
| 初回表示は idle 状態で API 未発火 | AC-1.1 |
| 1 文字以上入力 + 300ms 経過で API 1 回発火、loading 表示 | AC-1.2 |
| 連続入力中の debounce タイマーリセットと前リクエストの cancel | AC-1.3 |
| 結果 1 件以上で loaded、各行に必須フィールドが描画される | AC-1.4 |
| 結果 0 件で no-results 表示、キーワードが引用符で囲まれる | AC-1.5 |
| キーワード全消去で idle 復帰、API 未発火 | AC-1.6 |
| 編集 UI 起動・各 qualifier 編集 | AC-2.1, AC-2.2 |
| 編集中値変更で API 未発火 | AC-2.3 |
| 「適用」でチップ更新 + 再検索発火 | AC-2.4 |
| 「キャンセル」で編集中値破棄、適用済み不変 | AC-2.5 |

| 空クエリ + ソート変更のみで API 未発火 | AC-3.4 |
| 末尾インジケータ表示でページング発火 | AC-4.1 |

| 累積 1000 件で終端メッセージ + 以降未発火 | AC-4.4 |
| ページング失敗で既存結果保持 + 再試行ボタン | AC-4.5 |
