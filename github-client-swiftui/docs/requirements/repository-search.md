# リポジトリ検索 仕様書

本書はリポジトリ検索画面における **検索条件の構築** と **検索結果のソート** に関する仕様を定義する。`requirements.md` の「3.1 リポジトリ検索」を拡張する位置付けであり、本書と矛盾する場合は本書を優先する。

## 1. スコープ

### 1.1 対象

- リポジトリ検索画面の検索条件編集 UI
- GitHub Search API (`GET /search/repositories`) のクエリ文字列構築
- 検索結果のソート（API パラメータ `sort` / `order`）

### 1.2 対象外

- 表示モード（List / Grid 等）の切り替え
- 検索条件の永続保存・名前付きプリセット
- 検索履歴
- OR / NOT による条件式の入れ子
- カスタムプロパティ (`props.*`)
- リポジトリ詳細以降の画面

## 2. 検索条件

### 2.1 全体方針

- 検索条件は **キーワード** と **複数の qualifier** から構成される。
- GitHub Search API のクエリ言語では、空白区切りで並べた qualifier 同士はすべて AND として解釈される（同じ qualifier を複数回書いた場合も同様）。本書の「すべての条件は AND で結合される」はこの API 仕様に従う方針を意味する。
- 本フェーズでは OR / NOT は明示的にサポートしない。明示的な `OR` / `NOT` キーワードや先頭ハイフンによる除外もクエリ文字列には出力しない。
- 条件の編集は **専用の編集モーダル**（Sheet）で行い、「適用」を押した時点で検索画面側に確定済みクエリとして反映される。モーダル上で編集中の状態は確定前にはネットワークリクエストを発生させない。
- 検索画面の検索バー直下には適用中の条件をチップとして表示し、個別に × で削除できる。
- キーワード・qualifier いずれも空の場合は API を呼ばず、初期状態を表示する。

### 2.2 サポートする検索条件

#### 2.2.1 キーワード

- 単一のフリーテキスト。
- 複数語入力時は GitHub API の AND 解釈に従う。
- 入力には debounce を効かせる。debounce 仕様は `requirements.md` 5.4 に従う。

#### 2.2.2 検索対象 (`in:`)

- 検索キーワードを照合する対象を指定する。
- `name` / `description` / `readme` / `topics` から **複数選択可**。
- 未指定時は GitHub API のデフォルト動作に委ねる。

#### 2.2.3 言語 (`language:`)

- 単一指定。
- 候補はアプリ内に同梱した静的リストから選択する。GitHub には言語一覧を返す API が存在しないため、Linguist 由来の言語名から検索でよく使われるものを選定して同梱する。
- 候補リストは差し替え可能な抽象（`LanguageCatalog` 等）越しに参照し、Preview / Test で Mock に差し替えられる構造にする。
- 未指定時は条件として付与しない。

#### 2.2.4 スター数 (`stars:`)

- 最小値・最大値を指定する範囲フィルタ。
- 最小のみ、最大のみ、両方指定のいずれも許容する。
- API 文字列への変換ルール:
  - 最小のみ: `stars:>=N`
  - 最大のみ: `stars:<=N`
  - 両方: `stars:N..M`

#### 2.2.5 最終 push 日 (`pushed:`)

- 「指定日以降」「指定日以前」「範囲」のいずれかを指定する。
- 日付は ISO8601 (`YYYY-MM-DD`) で扱う。
- API 文字列への変換ルール:
  - 以降のみ: `pushed:>=YYYY-MM-DD`
  - 以前のみ: `pushed:<=YYYY-MM-DD`
  - 範囲: `pushed:YYYY-MM-DD..YYYY-MM-DD`

#### 2.2.6 トピック (`topic:`)

- 文字列指定。
- **複数指定可**で、それぞれ AND として連結する（例: `topic:ios topic:swiftui`）。

### 2.3 バリデーション

検索条件の Observable Model は以下の妥当性検証を行い、不正な間は「適用」操作を無効化する。エラー表示は条件単位で行う。

- 数値範囲（スター）: 最小 > 最大 は不正。負の数は不正。
- 日付範囲（pushed）: 開始 > 終了 は不正。未来日付は許容するが、API 側のレスポンスに従う。
- トピック・言語の文字列: 前後の空白はトリムし、空文字は条件として無視する。

### 2.4 クエリ文字列の構築

- 編集中の条件モデル（typed な値の集合）を確定時点で `q` パラメータの文字列にシリアライズする。
- シリアライズ責務は単一のコンポーネントに閉じ込め、Swift Testing で単独に検証可能とする。
- 同種の qualifier が複数許容される場合（例: `topic:`）は、それぞれを空白区切りで連結する。
- 値に空白を含む場合は `"..."` で囲む。
- 不正な条件はシリアライズ対象から除外する。

### 2.5 検索条件チップ

- 検索バー直下に「適用中の条件」を要約表示する。
- チップ単位で削除でき、削除すると即時に再検索が走る。
- キーワードが空かつチップが 0 件のときは何も表示しない。

## 3. ソート

### 3.1 ソートキー

以下のみをサポートする。

- `stars` — スター数
- `updated` — 最終更新日

API パラメータ `sort` に変換される。未指定時はベストマッチで、本フェーズではこの未指定状態は提供しない（必ずどちらかが選ばれている）。

初期値は `stars` とする。

### 3.2 並び順

- 昇順 / 降順を切り替え可能とする。API パラメータ `order` (`asc` / `desc`) に変換される。
- 初期値は降順とする。

### 3.3 ソート変更時の挙動

- ソートキー・並び順のいずれかが変更された場合、即時に API へ再リクエストを行う。
- 再リクエスト中は通常の loading 状態に戻し、前の結果はクリアする（並びだけ変わった結果が一瞬見える状態を避ける）。
- 進行中のリクエストは協調的にキャンセルする。`requirements.md` 5.3 のキャンセル方針に従う。

## 4. UI 仕様

### 4.1 検索画面

- 既存のキーワード入力欄を維持する。
- 検索バー右側に「条件編集を開く」ためのフィルタアイコンを配置する。
- 検索バー直下に適用中の条件チップ列を表示する。
- 画面右上にソートメニューを配置し、ソートキーと並び順を切り替えられるようにする。
- 結果リストの状態（loading / empty / error / loaded）は `requirements.md` 4 章に従う。

### 4.2 条件編集モーダル

- Sheet として表示する。`NavigationStack` の push にはしない。
- 条件種別ごとにセクションを設ける。
- フッターに「適用」「キャンセル」「すべてクリア」のアクションを置く。
- 「適用」は条件が妥当な間だけ有効。
- 「キャンセル」は編集中の draft を破棄して閉じる。
- 「すべてクリア」は draft を初期状態に戻す（モーダルは閉じない）。

### 4.3 ナビゲーション

- 条件編集モーダルは検索画面の route 配下に閉じた表示状態として扱う。
- typed route 配列には積まない。
- 確定済みクエリ・ソート設定は検索画面の Observable Model が保持する。

## 5. 状態管理

### 5.1 編集中状態と確定状態の分離

- 検索画面の Observable Model は **確定済みクエリ** と **ソート設定** を保持する。
- 条件編集モーダルの Observable Model は **編集中 draft** を保持し、確定済みクエリを初期値として複製してから編集を始める。
- 「適用」操作で draft を確定済みクエリに転写し、検索画面が再検索を発火する。
- 「キャンセル」操作で draft を破棄する。確定済みクエリには影響しない。

### 5.2 再検索の発火条件

以下のいずれかが変化したとき、検索画面は再検索を発火する。

- キーワード（debounce 経由）
- 確定済み qualifier 集合
- ソートキー
- 並び順

### 5.3 空クエリの扱い

- キーワードが空 かつ qualifier が 0 件のときは API を呼ばず、初期状態を表示する。
- ソートのみが選ばれている状態では検索を発火しない。

## 6. 実装方針

本章は仕様の意図を実装に落とすうえでの方針を示す。具体的な型名・ファイル名は実装着手時の合意で決定する余地を残すが、責務分離の方向性は本書で固定する。

### 6.1 レイヤ構成

`requirements.md` 5.1 のレイヤに加え、本機能では **Service 層** を新設する。Service 層は状態を持たない純粋ロジックを集約する層であり、Observable Model から呼び出される。

- View — 検索画面、条件編集モーダル
- Observable Model — 検索画面用、条件編集モーダル用
- Service 層（新設）
  - 検索条件 → クエリ文字列の構築
  - ソート設定 → API パラメータの変換
  - 言語候補リストの提供
- Repository — 検索 API 呼び出し
- API Client

### 6.2 ドメインモデル

- 検索条件全体は単一の値型で表現し、キーワード・検索対象 (`in:`) 集合・qualifier 列を保持する。
- qualifier は **sum type（enum）** として、種別ごとに必要な値を持つ associated value で表現する。これにより不正な状態を型レベルで排除し、シリアライズや UI 分岐を網羅的に書けるようにする。
- 数値範囲・日付範囲は最小/最大いずれか一方のみ指定も許容する独立した値型として扱う。
- 検索条件は `Sendable` を満たし、Task 境界を越えて受け渡せること。

### 6.3 状態管理

- 検索画面の Observable Model は **確定済みの検索条件**・ソート設定・結果状態を保持する。
- 条件編集モーダルの Observable Model は **編集中 draft** を保持し、確定済み検索条件を初期値として複製してから編集する。
- 「適用」操作で draft を確定済みに転写し、検索画面が再検索を発火する。
- 「キャンセル」操作で draft を破棄し、確定済みには触れない。

### 6.4 候補リストの取得

- 言語候補リストはアプリ内の静的リソース（JSON 等）から読み込む。
- 差し替え可能な抽象越しにアクセスし、Preview / Test では Mock を注入できる構造にする。

## 7. テスト要件

`requirements.md` 7 章に加え、以下を Swift Testing で検証可能にする。

- 各 qualifier の単独シリアライズ結果が正しい。
- 複数 qualifier を組み合わせたときの最終クエリ文字列が正しい。
- 数値範囲・日付範囲のバリデーションが境界値で正しく機能する。
- ソートキー・並び順の変更で API リクエストパラメータが期待通りに変換される。
- 編集中 draft の破棄が確定済みクエリに影響しない。
- チップ削除による条件除去が再検索を発火する。
