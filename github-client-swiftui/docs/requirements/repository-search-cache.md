# リポジトリ検索 結果キャッシュ 仕様書

> 本書はリポジトリ検索における「同一クエリの結果を短期間メモリに保持し、再リクエストを抑止する」機能の仕様を定義する。`requirements.md` の「その他」に挙げられている「パフォーマンス面での、同じクエリ検索の時間キャッシュ」を正式スコープに昇格させたもので、`docs/requirements/repository-search.md`（以下、本書では「検索 PRD」と呼ぶ）の関連 AC を拡張する位置付けである。本書と検索 PRD が矛盾する場合は **本書を優先** する。プロジェクト全体方針は `requirements.md` および `CLAUDE.md` を参照し、本書では再掲しない。

## 1. 目的と背景

- 同一クエリでの再検索（**入力 → 別キーワード → 戻し、ソート切替の往復、チップ削除と復元、詳細画面からの戻り** など）が頻繁に発生するが、現状は **そのたびに GitHub Search API へリクエスト** している。これは応答時間・通信量・レート制限 (Search API: unauthenticated 10 req/min) のいずれの面でも非効率である。
- 想定ユーザーは、検索条件を試行錯誤しながらリポジトリを探す iOS 利用者。「**`swift` → `swifty` まで打って戻して `swift`**」「ソートを stars と updated で見比べる」「詳細画面で確認してから検索結果に戻る」といった往復操作を想定する。
- 直近で取得した検索結果を **メモリ上に LRU キャッシュ** として保持し、キャッシュ命中時は **API を呼ばずに即時 loaded 状態を提示** する。これにより応答性が向上し、レート制限の消費も抑制できる。
- 鮮度回復（古くなったかもしれない結果を取り直す）は **Pull-to-refresh** によりユーザーが明示的に行う。**時間経過による自動失効 (TTL) は設けない**。GitHub リポジトリ検索の星数・新着頻度はユーザー操作中に大きく変動するものではなく、Pull-to-refresh で十分とする。

## 2. スコープ

### 2.1 対象

- リポジトリ検索（検索 PRD §3 で定義する初回検索とページング）の結果に対するメモリ LRU キャッシュ層の導入。
- キャッシュ命中時の UI 振る舞い（loading を挟まず即時 loaded に遷移する規則）。
- キャッシュキーの定義（クエリ文字列 + ソートキー + 並び順 + ページ番号）。
- キャッシュ容量上限と LRU 退避規則。
- キャッシュ無効化条件（プロセス終了 / 明示的なクリア操作）。
- Pull-to-refresh による明示的な再取得。

### 2.2 対象外

以下は本フェーズでは扱わない。§10 に将来課題として再掲する。

- 時間経過による自動失効 (TTL)。本書では採用しない（§1 の方針）。
- ディスク永続キャッシュ（プロセスをまたぐキャッシュ）。
- HTTP 層キャッシュ（`URLCache` ベース）。GitHub Search API は `Cache-Control` に依存できないため、本書はアプリ層メモリキャッシュとして規定する。
- 検索結果以外（リポジトリ詳細、Issue 一覧、Issue 詳細、コメント、アバター画像）のキャッシュ。
- stale-while-revalidate（古い結果を一旦見せながら裏で再取得する戦略）。
- in-flight request deduplication（同一クエリの同時実行リクエストを 1 つに統合する仕組み）。
- キャッシュ命中をユーザーに明示的に知らせる UI 表示（バッジ・取得時刻表示など）。
- キャッシュサイズ上限の細かな調整 UI（容量は内部で固定値）。
- キャッシュヒット率の計測・テレメトリ送出。

## 3. ユーザーストーリーと受け入れ条件

### US-1: 同じ条件で戻ったとき即時に結果を見たい

開発者として、一度検索した条件で再度結果を見るとき、待ち時間なしに直近の結果が表示されてほしい。

- AC-1.1 検索を実行し loaded 状態を得た後、**まったく同じキーワード・qualifier・ソート** で再検索が発火したとき、API リクエストは発行されず、画面は **loading 状態を経由せずに直接 loaded 状態** へ遷移する。表示される結果リストは前回 API レスポンス時と同一である。
- AC-1.2 上記のキャッシュ命中時、結果リストの内容（行の並び・件数・各行の表示項目）は前回 API レスポンスと完全に一致する。
- AC-1.3 キャッシュ命中時にユーザーから観測できる UI 表示は通常の loaded 状態と区別がつかない。「キャッシュ表示中」を示す追加 UI は表示しない。

### US-2: 文字列を行き来しても再取得しない

開発者として、`swift` → `swifty` → `swift` のように入力を行き来させたとき、最後に戻したキーワードはキャッシュから即時表示されてほしい。

- AC-2.1 入力中に複数の異なるクエリで検索が走った後、過去に取得済みのクエリへ戻したとき、容量上限（§5.4）に押し出されていない限りはキャッシュから即時 loaded 状態が再現される。API リクエストは発行されない。
- AC-2.2 キャッシュは時間経過では失効しない。プロセス終了 / 容量上限超過 / Pull-to-refresh のみが失効条件である。

### US-3: ページングもキャッシュから即時表示する

開発者として、検索結果一覧をスクロールしてページングしたあと別画面から戻ってきても、同じスクロール深度まで瞬時に復元したい。

- AC-3.1 ページング（検索 PRD §3 US-4）で取得した各ページは **ページ番号ごとに個別のキャッシュエントリ** として保存される。
- AC-3.2 詳細画面から戻ったあと、ユーザーが結果リスト末尾の追加読み込みインジケータまでスクロールしたとき、次ページ番号が **キャッシュ命中する** 場合は API を呼ばず、リスト末尾に loading インジケータを表示せずに、即時に新規アイテムが追記される。
- AC-3.3 次ページ番号が **キャッシュ未登録（押し出し済み含む）** の場合は、通常のページングフロー（リスト末尾 loading → API → 追記）に従う。
- AC-3.4 page=1 と page=2 が混在するケース（例: page=1 はキャッシュ命中、page=2 は押し出されている）でも、それぞれのページ単位で独立に命中判定が行われる。

### US-4: ソート切替時もキャッシュを活用する

開発者として、ソート方法を切り替えて見比べたあと、最初のソートに戻したとき、待ち時間なしで元の結果に戻りたい。

- AC-4.1 同じキーワード・qualifier で **ソートキーまたは並び順だけが異なる** 検索は、それぞれ独立したキャッシュエントリとして扱われる。
- AC-4.2 ソートを切り替えて元のソートへ戻したとき、元のソートのキャッシュエントリが残っていればキャッシュ命中の振る舞い（AC-1.1）が適用される。

### US-5: 異常系はキャッシュ対象外

開発者として、エラー結果が一定時間使い回されることなく、再試行のたびに最新状態を確認したい。

- AC-5.1 error-network / error-rate-limited 状態に遷移したレスポンスは **キャッシュに保存しない**。
- AC-5.2 no-results 状態（API レスポンス成功で 0 件）は **キャッシュに保存する**（loaded と同じく成功応答であるため）。同条件の再検索は API を呼ばず no-results を即時表示する。
- AC-5.3 ユーザー操作起因のキャンセル（検索 PRD §4.3.3）で中断されたリクエストの結果はキャッシュに保存しない。

### US-6: 鮮度を取り戻したい

開発者として、検索結果が古い気がしたら明示的に最新の結果を取得し直したい。

- AC-6.1 検索結果リストを画面下方向に引いて離す操作（Pull-to-refresh）により、**現在の検索条件（クエリ + ソート）の page=1 のキャッシュを破棄** し、API から再取得する。
- AC-6.2 Pull-to-refresh で再取得した結果は新しい LRU エントリとして再登録される。同じ条件の page=2 以降のキャッシュも破棄され、ユーザーがスクロールしたタイミングで再取得される（AC-3.3 と同じ流れに合流する）。
- AC-6.3 Pull-to-refresh 中は通常の loading インジケータ（標準の SwiftUI `.refreshable` の挙動）を表示する。

### US-7: アプリのライフサイクルとキャッシュ

開発者として、アプリを再起動したあとは新鮮な結果から始めたい。

- AC-7.1 アプリプロセスが終了したとき、キャッシュはすべて失われる（メモリのみ保持、永続化しない）。
- AC-7.2 アプリがバックグラウンド遷移してもメモリ上のキャッシュは保持される。フォアグラウンド復帰時の追加処理は行わない。
- AC-7.3 検索画面を `NavigationStack` で pop した（戻る操作で破棄された）場合も、検索画面が再構築されたときに同じキャッシュを利用できる。すなわち **キャッシュは検索画面のライフサイクルとは独立** に保持される。

## 4. 振る舞い仕様

### 4.1 入出力一覧

検索 PRD §4.1 の入出力一覧に対し、本機能で振る舞いが変わる行のみを示す。記載のない操作は検索 PRD の規定どおり。

| ユーザー操作 / 入力 | 前提条件 | システム応答 / 表示 |
|---|---|---|
| キーワード入力 → debounce 経過で再検索発火 | 同条件・同ソート・page=1 のキャッシュが残っている | API 未発火。loading 状態を経由せず即時に loaded（または no-results）へ遷移 |
| キーワード入力 → debounce 経過で再検索発火 | 該当キャッシュなし | 検索 PRD §4.1 のとおり。loading 表示 → API 発火 → 結果反映後にキャッシュ保存 |
| 編集 UI で「適用」 | 適用後の条件・ソート・page=1 のキャッシュが残っている | 編集 UI が閉じ、loading を経由せず即時 loaded / no-results 表示。チップ列のみ更新 |
| 編集 UI で「適用」 | 該当キャッシュなし | 検索 PRD §4.1 のとおり |
| チップ × タップ | 削除後の条件・ソート・page=1 のキャッシュが残っている | API 未発火。即時 loaded / no-results 表示 |
| チップ × タップ | 該当キャッシュなし | 検索 PRD §4.1 のとおり |
| ソートキーまたは並び順を変更 | 変更後のソートでの page=1 キャッシュが残っている | API 未発火。loading を経由せず即時 loaded / no-results 表示 |
| ソート変更 | 該当キャッシュなし | 検索 PRD §4.1 のとおり |
| 結果末尾のローディングが画面に表示 | 次ページ番号のキャッシュが残っている | API 未発火。リスト末尾の loading インジケータを表示せず即時に新規アイテム追記 |
| 結果末尾のローディングが画面に表示 | 次ページ番号のキャッシュなし、かつ累積 < 1000 | 検索 PRD §4.1 のとおり。API 発火後にキャッシュ保存 |
| 結果リストで Pull-to-refresh | loaded 状態 | 現在条件の page=1 以降のキャッシュを破棄し、page=1 を API から再取得 |
| 再試行ボタンタップ（初回検索の error 状態から） | 異常系遷移後 | API 発火。前回の error 状態自体はキャッシュ対象外（AC-5.1） |
| 検索画面 pop → 同じ条件で検索画面再構築 | 同条件のキャッシュが残っている | 再構築直後の初回検索発火時に API 未発火、即時 loaded |
| アプリ再起動 | — | プロセス再起動でメモリキャッシュは消失。初回検索は通常フローで発火 |

### 4.2 状態と表示

検索 PRD §4.2 の 8 状態のうち、キャッシュ命中時は **loading を介さず直接** 以下のいずれかへ遷移する点だけが本機能で変わる。

- 初回検索（page=1）のキャッシュ命中時: 直前の状態（idle / loaded / no-results / error-* / paging-*）から **loading を経由せず** に `loaded` または `no-results` へ遷移する。
- ページング（page>=2）のキャッシュ命中時: `loaded` 状態のまま、リスト末尾に paging-loading 表示を出さずに、リストへ新規アイテムが追記される。

その他の状態定義および遷移トリガは検索 PRD §4.2 と同じ。「キャッシュ命中中であること」を示す独立した状態は導入しない（AC-1.3）。

### 4.3 バリデーションと異常系

- 入力バリデーション・エラー分類は検索 PRD §4.3 のとおり。本機能で追加のバリデーションは発生しない。
- キャッシュエントリそのもののバリデーション（キー一致と存在判定）は §5 で規定する。

## 5. キャッシュ仕様

### 5.1 キャッシュキー

キャッシュキーは以下の 3 要素のタプルで一意に定まる。

| 要素 | 値の形 | 由来 |
|---|---|---|
| `q` | クエリ文字列（検索 PRD §5.1〜§5.2 の規則で構築済み） | 検索 PRD §5.1 で「並びを決定的」と規定されているため、同条件は常に同一文字列になる |
| `sort` | `RepositorySearchSort`（key + order を内包する Hashable 型） | 検索 PRD §5.4。ソートキー (stars/updated) と並び順 (asc/desc) を含む |
| `page` | 整数（1 以上） | 検索 PRD §5.4 |

3 要素のいずれか 1 つでも異なれば別エントリとして扱う。

### 5.2 キャッシュエントリの内容

各キャッシュエントリは API レスポンス本体のみを保持する。

- レスポンス本体: API レスポンスから構築された結果（loaded の場合は GitHub リポジトリの配列、no-results の場合は「結果 0 件」を示す状態）。検索結果以外の付随情報（GitHub Search API の `total_count` 等で UI 上意味のあるもの）も保持する。具体的な内部構造は実装に委ねるが、**ユーザーから観測される行情報・件数情報・並び順は API レスポンス時点と同一に再現できる** こと。
- 上記以外の付加情報（取得元 URL、ヘッダ、取得時刻等）は保存しない。

### 5.3 ライフタイム

- 失効条件は以下のみ:
  1. プロセス終了
  2. 容量上限超過時の LRU 退避（§5.4）
  3. Pull-to-refresh による明示破棄（§5.7）
- 時間経過による自動失効 (TTL) は **設けない**（§1）。
- 失効したエントリはアクセス時に未登録として扱い、通常の API 発火フローに復帰する。失効したことをユーザーに知らせる UI は出さない。
- 永続化なし。プロセス終了でキャッシュは消える（AC-7.1）。

### 5.4 キャッシュサイズと退避規則

- 同時に保持できるエントリ数の上限は **100 件**（page 単位）。
- 上限到達時は **最も古くアクセスされていないエントリ (LRU) から退避** する。具体的には:
  - エントリの **書き込み（put）** または **読み出し成功（get hit）** をアクセスとみなす。
  - 上限を超えた `put` 時に、アクセス順で最も古いエントリを 1 件以上退避する。
- このサイズ上限はメモリ保護のための安全弁であり、通常の検索 → ページング → 別条件検索程度では到達しないことを想定する。

### 5.5 キャッシュ書き込みのタイミング

- API レスポンスが成功し、結果が loaded または no-results として確定した直後に、対応するキー（§5.1）のエントリを書き込む。
- error-network / error-rate-limited 状態に遷移したレスポンスは書き込まない（AC-5.1）。
- ユーザー操作起因のキャンセル（CancellationError）で中断されたリクエストは書き込まない（AC-5.3）。

### 5.6 キャッシュ参照のタイミング

検索発火（初回検索 / ページング）の直前に、対応するキー（§5.1）でキャッシュを参照する。命中したらキャッシュエントリの内容で UI 状態を確定し、API は呼ばない。未登録なら通常のフローに進む。

### 5.7 Pull-to-refresh による明示破棄

- リスト上で Pull-to-refresh が実行されたとき、**現在のクエリ + ソートに紐づく page=1 以上の全エントリ** をキャッシュから破棄し、page=1 から API 再取得を行う。
- 破棄対象は「現在条件と同一 `q + sort` を持つ全ページ」。他のクエリ・ソートのエントリは影響を受けない。
- 再取得した結果は新たに LRU エントリとして登録される。

## 6. 画面構成

本機能はバックグラウンド最適化と Pull-to-refresh 操作の追加。新規画面の追加・既存画面のレイアウト変更は伴わない。

- 検索画面（検索 PRD §5）に「キャッシュ表示中」を示す UI 要素は追加しない（AC-1.3）。
- 検索結果リストに **Pull-to-refresh ジェスチャ** を追加する。SwiftUI 標準の `.refreshable` を使い、固有のスタイリングは付けない。
- 検索画面の操作要素・配置・状態ごとの見え方は検索 PRD §5 のままとする。
- ユーザーから観測できる変化は「同条件再検索時に loading 表示が出なくなる」「ページング時に末尾 loading が出なくなる」「リスト上で Pull-to-refresh ができる」の 3 点。

## 7. データ要件

### 7.1 外部データソース

- 検索 PRD §6.1 のとおり（GitHub Search API `GET /search/repositories`）。本機能は API 呼び出し自体を変更しない（呼び出すかどうかの前段でキャッシュ判定するのみ）。

### 7.2 必要なフィールド

- キャッシュエントリが保持するフィールドは、API レスポンスから loaded / no-results 状態を再現できる範囲に限定する。
- ユーザーから観測できない内部メタ情報（HTTP ヘッダ、URLResponse 等）は保持しない。

### 7.3 制約

- 検索 PRD §6.3 の制約（ページング上限 1000 件、per_page=30 等）はそのまま適用される。本機能で新たな外部制約は発生しない。

## 8. 非機能要件

### 8.1 性能

- キャッシュ命中時、ユーザー操作（検索バー入力確定、チップ削除、ソート変更、適用、ページング末尾到達）から `loaded` / `no-results` 状態の画面反映までの時間は **p95 で 100ms 以内**。
- キャッシュ参照および書き込み 1 回あたりの所要時間は **10ms 以内**（端末上でのローカル処理として）。
- キャッシュ命中によって既存性能要件（検索 PRD §8.1 の API レスポンス p95 1.5 秒以内、ローカル処理 100ms 以内）が劣化しないこと。

### 8.2 メモリ

- キャッシュ全体（最大 100 エントリ）の保持メモリは **アプリ全体のメモリプロファイル上、検索画面表示中の baseline に対し著しく増えないこと**。具体的なバイト数の上限は実装側の計測で決める（仕様としては「100 エントリの上限が守られていれば良い」とする）。

### 8.3 アクセシビリティ

- キャッシュ命中時の loaded / no-results 表示は通常時と完全に同じ表示・同じアクセシビリティ属性とする（VoiceOver の読み上げ・Dynamic Type への追従が変わらない）。
- Pull-to-refresh ジェスチャ自体は SwiftUI 標準の `.refreshable` がアクセシビリティに対応するため、追加の配慮は不要。

### 8.4 ローカライズ

- 本機能はユーザー可視文言を追加しない（キャッシュ表示中のサインを出さない、Pull-to-refresh のラベルは SwiftUI 標準）。検索 PRD §8.3 と同じ。

## 9. 完了の定義

以下のすべてを満たしたとき、本機能は完了とみなす。

- §3 のすべての受け入れ条件（AC-1.1〜AC-7.3）が満たされる。
- §4 の入出力表と状態遷移が仕様どおりに動作する。
- §5 のキャッシュキー・LRU 容量上限・書き込み / 参照タイミング・Pull-to-refresh による破棄がすべて仕様どおりに動作する。
- §10 の検証要件のテストが Swift Testing ですべて通る。
- §8 の非機能目標を満たす（キャッシュ命中時の p95 100ms、検索 PRD §8.1 の既存性能目標を劣化させない）。
- 既存の検索 PRD §3 のすべての AC が、本機能導入後も引き続き満たされる（loading を挟むケースの条件分岐が崩れていないこと）。

## 10. 検証要件と AC の対応

§3 の各受け入れ条件に対し、Swift Testing で検証すべき観点を定義する。具体的なテストコードや Mock 構造には踏み込まない。

| 検証観点 | 対応 AC |
|---|---|
| 同条件・同ソート・page=1 の再検索で API 未発火、loading を経由せず loaded 即時遷移 | AC-1.1 |
| キャッシュ命中時の結果リストが直前の API レスポンスと完全一致 | AC-1.2 |
| キャッシュ命中時に追加 UI 要素が表示されないこと | AC-1.3 |
| 「q1 → q2 → q1」の往復で q1 がキャッシュから即時再現されること（容量上限内） | AC-2.1 |
| 時間経過のみではキャッシュが失効しないこと（プロセス継続中は残る） | AC-2.2 |
| page ごとに独立したキャッシュエントリが作られ命中判定されること | AC-3.1 |
| キャッシュ命中ページングで末尾 loading を経由せず即時追記されること | AC-3.2 |
| 次ページが未登録のときは通常のページングフローに復帰すること | AC-3.3 |
| page=1 命中 + page=2 未登録の混在ケースで個別判定が機能すること | AC-3.4 |
| 同条件でソートだけ異なる検索が独立エントリとして扱われること | AC-4.1 |
| ソート往復で元のソートのキャッシュが命中復帰すること | AC-4.2 |
| error-network / error-rate-limited 状態はキャッシュに保存されないこと | AC-5.1 |
| no-results 状態はキャッシュに保存され再検索で即時再現されること | AC-5.2 |
| キャンセルされたリクエストの結果はキャッシュに保存されないこと | AC-5.3 |
| Pull-to-refresh で現在条件の page=1 以降が破棄され、API から再取得されること | AC-6.1, AC-6.2 |
| バックグラウンド遷移を理由にキャッシュが破棄されないこと | AC-7.2 |
| 検索画面 pop → 再構築でも同条件キャッシュが再利用されること | AC-7.3 |
| キャッシュキーのタプル一意性（q / sort / page のいずれか 1 つ違いで別エントリ） | §5.1 |
| キャッシュエントリ 100 件超過時に LRU で最も古くアクセスされていないエントリから破棄されること | §5.4 |
| `get` 成功でアクセス順が更新され、その後の `put` で別エントリが優先退避されること | §5.4 |
| キャッシュ命中時のユーザー操作から画面反映まで p95 100ms 以内（自動測定可能な範囲で） | §8.1 |

## 11. 将来課題

§2.2 で除外した項目のうち、優先度の高い順に挙げる。

1. stale-while-revalidate（古い結果を即時表示しながら裏で再取得し、差し替える）。
2. in-flight request deduplication（同一クエリの同時実行リクエスト統合）。
3. リポジトリ詳細 / Issue 一覧 / Issue 詳細 / コメントのキャッシュ。
4. ディスク永続キャッシュとプロセス間共有。
5. HTTP 層キャッシュ（`URLCache`）の活用検討。
6. キャッシュヒット率テレメトリ。
7. 時間経過による自動失効 (TTL) の再導入。データ鮮度が問題になるケースが観測された場合に検討する。
