# リポジトリ検索 結果キャッシュ 仕様書

> 本書はリポジトリ検索における「同一クエリの結果を短期間メモリに保持し、再リクエストを抑止する」機能の仕様を定義する。`requirements.md` の「その他」に挙げられている「パフォーマンス面での、同じクエリ検索の時間キャッシュ」を正式スコープに昇格させたもので、`docs/requirements/repository-search.md`（以下、本書では「検索 PRD」と呼ぶ）の関連 AC を拡張する位置付けである。本書と検索 PRD が矛盾する場合は **本書を優先** する。プロジェクト全体方針は `requirements.md` および `CLAUDE.md` を参照し、本書では再掲しない。

## 1. 目的と背景

- 同一クエリでの再検索（ソート切替の往復、チップ削除と復元、詳細画面からの戻り、入力 → 戻し → 同じ入力 など）が頻繁に発生するが、現状は **そのたびに GitHub Search API へリクエスト** している。これは応答時間・通信量・レート制限 (Search API: unauthenticated 10 req/min) のいずれの面でも非効率である。
- 想定ユーザーは、検索条件を試行錯誤しながらリポジトリを探す iOS 利用者。「ソートを stars と updated で見比べる」「詳細画面で確認してから検索結果に戻る」「条件を 1 つ変えてまた戻す」といった往復操作を想定する。
- 直近の検索結果を **TTL 60 秒のメモリキャッシュ** で保持し、キャッシュ命中時は **API を呼ばずに即時 loaded 状態を提示** する。これにより応答性が向上し、レート制限の消費も抑制できる。
- 60 秒という短い TTL は、結果鮮度の劣化に対するユーザー体感影響を小さく抑える狙いである。

## 2. スコープ

### 2.1 対象

- リポジトリ検索（検索 PRD §3 で定義する初回検索とページング）の結果に対するメモリキャッシュ層の導入。
- キャッシュ命中時の UI 振る舞い（loading を挟まず即時 loaded に遷移する規則）。
- キャッシュキーの定義（クエリ文字列 + ソートキー + 並び順 + ページ番号）。
- キャッシュ TTL（60 秒、固定）と有効期限切れ時の振る舞い。
- キャッシュ無効化条件（プロセス終了 / アプリのバックグラウンド長期化 / 明示的なクリア操作の有無）。

### 2.2 対象外

以下は本フェーズでは扱わない。§10 に将来課題として再掲する。

- ディスク永続キャッシュ（プロセスをまたぐキャッシュ）。
- HTTP 層キャッシュ（`URLCache` ベース）。GitHub Search API は `Cache-Control` に依存できないため、本書はアプリ層メモリキャッシュとして規定する。
- 検索結果以外（リポジトリ詳細、Issue 一覧、Issue 詳細、コメント、アバター画像）のキャッシュ。
- stale-while-revalidate（古い結果を一旦見せながら裏で再取得する戦略）。
- in-flight request deduplication（同一クエリの同時実行リクエストを 1 つに統合する仕組み）。
- キャッシュ命中をユーザーに明示的に知らせる UI 表示（バッジ・取得時刻表示など）。
- 手動でキャッシュをクリアする UI 操作。
- キャッシュサイズ上限の細かな調整 UI（容量は内部で固定値）。
- キャッシュヒット率の計測・テレメトリ送出。

## 3. ユーザーストーリーと受け入れ条件

### US-1: 同じ条件で戻ったとき即時に結果を見たい

開発者として、一度検索した条件で再度結果を見るとき、待ち時間なしに直近の結果が表示されてほしい。

- AC-1.1 検索を実行し loaded 状態を得た直後（60 秒以内）に、**まったく同じキーワード・qualifier・ソート** で再検索が発火したとき、API リクエストは発行されず、画面は **loading 状態を経由せずに直接 loaded 状態** へ遷移する。表示される結果リストは直前の loaded 状態と同一である。
- AC-1.2 上記のキャッシュ命中時、結果リストの内容（行の並び・件数・各行の表示項目）は前回 API レスポンスと完全に一致する。
- AC-1.3 キャッシュ命中時にユーザーから観測できる UI 表示は通常の loaded 状態と区別がつかない。「キャッシュ表示中」を示す追加 UI は表示しない。

### US-2: 60 秒経ったら新しい結果を取り直す

開発者として、キャッシュが古くなったタイミングでは最新の検索結果を取得し直したい。

- AC-2.1 検索結果をキャッシュしてから **60 秒を超えて** から同条件で再検索が発火したとき、キャッシュは破棄され、通常の検索フロー（loading 表示 → API 発火 → loaded / no-results / error）が実行される。
- AC-2.2 TTL 起算点は **API レスポンスを受信し loaded 状態が成立した時刻**。検索発火時刻やリクエスト送出時刻ではない。
- AC-2.3 ちょうど 60 秒の境界では「`stored_at` から経過時間が `60秒以下`」をキャッシュ有効と判定する（`<= 60` 秒は命中、`> 60` 秒は失効）。

### US-3: ページングもキャッシュから即時表示する

開発者として、検索結果一覧をスクロールしてページングしたあと別画面から戻ってきても、同じスクロール深度まで瞬時に復元したい。

- AC-3.1 ページング（検索 PRD §3 US-4）で取得した各ページは **ページ番号ごとに個別のキャッシュエントリ** として保存される。
- AC-3.2 詳細画面から戻ったあと、ユーザーが結果リスト末尾の追加読み込みインジケータまでスクロールしたとき、次ページ番号が **キャッシュ命中する** 場合は API を呼ばず、リスト末尾に loading インジケータを表示せずに、即時に新規アイテムが追記される。
- AC-3.3 次ページ番号が **キャッシュ失効または未取得** の場合は、通常のページングフロー（リスト末尾 loading → API → 追記）に従う。
- AC-3.4 page=1 と page=2 が混在するケース（例: page=1 はキャッシュ命中、page=2 はキャッシュ失効）でも、それぞれのページ単位で独立に命中判定が行われる。

### US-4: ソート切替時もキャッシュを活用する

開発者として、ソート方法を切り替えて見比べたあと、最初のソートに戻したとき、待ち時間なしで元の結果に戻りたい。

- AC-4.1 同じキーワード・qualifier で **ソートキーまたは並び順だけが異なる** 検索は、それぞれ独立したキャッシュエントリとして扱われる。
- AC-4.2 ソートを切り替えて 60 秒以内に元のソートへ戻したとき、元のソートのキャッシュエントリが有効ならキャッシュ命中の振る舞い（AC-1.1）が適用される。

### US-5: 異常系はキャッシュ対象外

開発者として、エラー結果が一定時間使い回されることなく、再試行のたびに最新状態を確認したい。

- AC-5.1 error-network / error-rate-limited 状態に遷移したレスポンスは **キャッシュに保存しない**。
- AC-5.2 no-results 状態（API レスポンス成功で 0 件）は **キャッシュに保存する**（loaded と同じく成功応答であるため）。同条件の再検索は API を呼ばず no-results を即時表示する。
- AC-5.3 ユーザー操作起因のキャンセル（検索 PRD §4.3.3）で中断されたリクエストの結果はキャッシュに保存しない。

### US-6: アプリのライフサイクルとキャッシュ

開発者として、アプリを再起動したり長時間放置したりしたあとは、新鮮な結果から始めたい。

- AC-6.1 アプリプロセスが終了したとき、キャッシュはすべて失われる（メモリのみ保持、永続化しない）。
- AC-6.2 アプリがバックグラウンド遷移してもメモリ上のキャッシュは保持される。フォアグラウンド復帰時の追加処理は行わない（TTL 判定は §2.2 の規則のみで決まる）。
- AC-6.3 検索画面を `NavigationStack` で pop した（戻る操作で破棄された）場合も、検索画面が再構築されたときに同じキャッシュを利用できる。すなわち **キャッシュは検索画面のライフサイクルとは独立** に保持される。

## 4. 振る舞い仕様

### 4.1 入出力一覧

検索 PRD §4.1 の入出力一覧に対し、本機能で振る舞いが変わる行のみを示す。記載のない操作は検索 PRD の規定どおり。

| ユーザー操作 / 入力 | 前提条件 | システム応答 / 表示 |
|---|---|---|
| キーワード入力 → debounce 経過で再検索発火 | 同条件・同ソート・page=1 のキャッシュが有効（60 秒以内） | API 未発火。loading 状態を経由せず即時に loaded（または no-results）へ遷移 |
| キーワード入力 → debounce 経過で再検索発火 | 該当キャッシュなし or 失効 | 検索 PRD §4.1 のとおり。loading 表示 → API 発火 → 結果反映後にキャッシュ保存 |
| 編集 UI で「適用」 | 適用後の条件・ソート・page=1 のキャッシュが有効 | 編集 UI が閉じ、loading を経由せず即時 loaded / no-results 表示。チップ列のみ更新 |
| 編集 UI で「適用」 | 該当キャッシュなし or 失効 | 検索 PRD §4.1 のとおり |
| チップ × タップ | 削除後の条件・ソート・page=1 のキャッシュが有効 | API 未発火。即時 loaded / no-results 表示 |
| チップ × タップ | 該当キャッシュなし or 失効 | 検索 PRD §4.1 のとおり |
| ソートキーまたは並び順を変更 | 変更後のソートでの page=1 キャッシュが有効 | API 未発火。loading を経由せず即時 loaded / no-results 表示 |
| ソート変更 | 該当キャッシュなし or 失効 | 検索 PRD §4.1 のとおり |
| 結果末尾のローディングが画面に表示 | 次ページ番号のキャッシュが有効 | API 未発火。リスト末尾の loading インジケータを表示せず即時に新規アイテム追記 |
| 結果末尾のローディングが画面に表示 | 次ページ番号のキャッシュなし or 失効、かつ累積 < 1000 | 検索 PRD §4.1 のとおり。API 発火後にキャッシュ保存 |
| 再試行ボタンタップ（初回検索の error 状態から） | 異常系遷移後 | API 発火。前回の error 状態自体はキャッシュ対象外（AC-5.1） |
| 検索画面 pop → 同じ条件で検索画面再構築 | 同条件のキャッシュが有効 | 再構築直後の初回検索発火時に API 未発火、即時 loaded |
| アプリのバックグラウンド遷移 → 60 秒以内に復帰 | キャッシュあり | TTL 判定のみで通常の発火判定。バックグラウンド遷移を理由としたキャッシュ破棄は発生しない |
| アプリ再起動 | — | プロセス再起動でメモリキャッシュは消失。初回検索は通常フローで発火 |

### 4.2 状態と表示

検索 PRD §4.2 の 8 状態のうち、キャッシュ命中時は **loading を介さず直接** 以下のいずれかへ遷移する点だけが本機能で変わる。

- 初回検索（page=1）のキャッシュ命中時: 直前の状態（idle / loaded / no-results / error-* / paging-*）から **loading を経由せず** に `loaded` または `no-results` へ遷移する。
- ページング（page>=2）のキャッシュ命中時: `loaded` 状態のまま、リスト末尾に paging-loading 表示を出さずに、リストへ新規アイテムが追記される。すなわち `loaded → loaded`（リスト末尾に追記）の遷移を、ページング loading 状態を経由せずに行う。

その他の状態定義および遷移トリガは検索 PRD §4.2 と同じ。「キャッシュ命中中であること」を示す独立した状態は導入しない（AC-1.3）。

### 4.3 バリデーションと異常系

- 入力バリデーション・エラー分類は検索 PRD §4.3 のとおり。本機能で追加のバリデーションは発生しない。
- キャッシュエントリそのもののバリデーション（TTL 判定・キー一致）は §5 で規定する。

## 5. キャッシュ仕様

### 5.1 キャッシュキー

キャッシュキーは以下の 4 要素のタプルで一意に定まる。

| 要素 | 値の形 | 由来 |
|---|---|---|
| `q` | クエリ文字列（検索 PRD §5.1〜§5.2 の規則で構築済み） | 検索 PRD §5.1 で「並びを決定的」と規定されているため、同条件は常に同一文字列になる |
| `sort` | `stars` または `updated` | 検索 PRD §5.4 |
| `order` | `asc` または `desc` | 検索 PRD §5.4 |
| `page` | 整数（1 以上） | 検索 PRD §5.4 |

4 要素のいずれか 1 つでも異なれば別エントリとして扱う。

### 5.2 キャッシュエントリの内容

各キャッシュエントリは以下の情報を保持する。

- レスポンス本体: API レスポンスから構築された結果（loaded の場合は GitHub リポジトリの配列、no-results の場合は「結果 0 件」を示す状態）。検索結果以外の付随情報（GitHub Search API の `total_count` 等で UI 上意味のあるもの）も保持する。具体的な内部構造は実装に委ねるが、**ユーザーから観測される行情報・件数情報・並び順は API レスポンス時点と同一に再現できる** こと。
- 格納時刻 `stored_at`: API レスポンス受信し loaded / no-results 状態に確定した時点の時刻。
- 上記以外の付加情報（取得元 URL、ヘッダ等）は保存しない。

### 5.3 TTL とライフタイム

- TTL: **60 秒**。判定式は「現在時刻 − `stored_at` ≦ 60 秒なら有効、超過したら失効」。
- 時刻基準: 端末ローカル時刻。NTP 同期等の特殊配慮は不要。
- 失効したエントリはアクセス時に **静かに破棄** され、通常の API 発火フローに復帰する。失効したことをユーザーに知らせる UI は出さない。
- 永続化なし。プロセス終了でキャッシュは消える（AC-6.1）。

### 5.4 キャッシュサイズ

- 同時に保持できるエントリ数の上限は **100 件**（page 単位）。
- 上限到達時は **最も古い `stored_at` を持つエントリから順に破棄** する（FIFO ベース、stored_at の昇順）。
- このサイズ上限はメモリ保護のための安全弁であり、通常の検索 → ページング → 別条件検索程度では到達しないことを想定する。

### 5.5 キャッシュ書き込みのタイミング

- API レスポンスが成功し、結果が loaded または no-results として確定した直後に、対応するキー（§5.1）のエントリを書き込む。
- error-network / error-rate-limited 状態に遷移したレスポンスは書き込まない（AC-5.1）。
- ユーザー操作起因のキャンセル（CancellationError）で中断されたリクエストは書き込まない（AC-5.3）。

### 5.6 キャッシュ参照のタイミング

検索発火（初回検索 / ページング）の直前に、対応するキー（§5.1）でキャッシュを参照する。命中したらキャッシュエントリの内容で UI 状態を確定し、API は呼ばない。失効・未登録なら通常のフローに進む。

### 5.7 キャッシュの明示的クリア

本フェーズではキャッシュを明示的にクリアする UI 操作は提供しない（§2.2）。TTL 失効とプロセス終了のみでクリアされる。

## 6. 画面構成

本機能はバックグラウンド最適化であり、新規画面の追加・既存画面のレイアウト変更を伴わない。

- 検索画面（検索 PRD §5）に「キャッシュ表示中」を示す UI 要素は追加しない（AC-1.3）。
- 検索画面の操作要素・配置・状態ごとの見え方は検索 PRD §5 のままとする。
- ユーザーから観測できる変化は「同条件再検索時に loading 表示が出なくなる」「ページング時に末尾 loading が出なくなる」の 2 点のみ。

## 7. データ要件

### 7.1 外部データソース

- 検索 PRD §6.1 のとおり（GitHub Search API `GET /search/repositories`）。本機能は API 呼び出し自体を変更しない（呼び出すかどうかの前段でキャッシュ判定するのみ）。

### 7.2 必要なフィールド

- キャッシュエントリが保持するフィールドは、API レスポンスから loaded / no-results 状態を再現できる範囲に限定する。
- ユーザーから観測できない内部メタ情報（HTTP ヘッダ、URLResponse 等）は保持しない。

### 7.3 制約

- 検索 PRD §6.3 の制約（ページング上限 1000 件、per_page=30 等）はそのまま適用される。本機能で新たな外部制約は発生しない。

## 8. 非機能要件

### 8.1 性能

- キャッシュ命中時、ユーザー操作（検索バー入力確定、チップ削除、ソート変更、適用、ページング末尾到達）から `loaded` / `no-results` 状態の画面反映までの時間は **p95 で 100ms 以内**。
- キャッシュ参照および書き込み 1 回あたりの所要時間は **10ms 以内**（端末上でのローカル処理として）。
- キャッシュ命中によって既存性能要件（検索 PRD §8.1 の API レスポンス p95 1.5 秒以内、ローカル処理 100ms 以内）が劣化しないこと。

### 8.2 メモリ

- キャッシュ全体（最大 100 エントリ）の保持メモリは **アプリ全体のメモリプロファイル上、検索画面表示中の baseline に対し著しく増えないこと**。具体的なバイト数の上限は実装側の計測で決める（仕様としては「100 エントリの上限が守られていれば良い」とする）。

### 8.3 アクセシビリティ

- キャッシュ命中時の loaded / no-results 表示は通常時と完全に同じ表示・同じアクセシビリティ属性とする（VoiceOver の読み上げ・Dynamic Type への追従が変わらない）。

### 8.4 ローカライズ

- 本機能はユーザー可視文言を追加しない（キャッシュ表示中のサインを出さないため）。検索 PRD §8.3 と同じ。

## 9. 完了の定義

以下のすべてを満たしたとき、本機能は完了とみなす。

- §3 のすべての受け入れ条件（AC-1.1〜AC-6.3）が満たされる。
- §4 の入出力表と状態遷移が仕様どおりに動作する。
- §5 のキャッシュキー・TTL・サイズ上限・書き込み / 参照タイミングがすべて仕様どおりに動作する。
- §10 の検証要件のテストが Swift Testing ですべて通る。
- §8 の非機能目標を満たす（キャッシュ命中時の p95 100ms、検索 PRD §8.1 の既存性能目標を劣化させない）。
- 既存の検索 PRD §3 のすべての AC が、本機能導入後も引き続き満たされる（loading を挟むケースの条件分岐が崩れていないこと）。

## 10. 検証要件と AC の対応

§3 の各受け入れ条件に対し、Swift Testing で検証すべき観点を定義する。具体的なテストコードや Mock 構造には踏み込まない。

| 検証観点 | 対応 AC |
|---|---|
| 同条件・同ソート・page=1 の再検索で API 未発火、loading を経由せず loaded 即時遷移 | AC-1.1 |
| キャッシュ命中時の結果リストが直前の API レスポンスと完全一致 | AC-1.2 |
| キャッシュ命中時に追加 UI 要素が表示されないこと | AC-1.3 |
| 61 秒経過後の同条件再検索で API 発火、通常フロー（loading 表示）が走ること | AC-2.1 |
| TTL 起算点が API レスポンス受信時刻であること（検索発火時刻ではない） | AC-2.2 |
| TTL 境界（60 秒ちょうど）が有効と判定されること、60.001 秒で失効すること | AC-2.3 |
| page ごとに独立したキャッシュエントリが作られ命中判定されること | AC-3.1 |
| キャッシュ命中ページングで末尾 loading を経由せず即時追記されること | AC-3.2 |
| 次ページが未取得 / 失効のときは通常のページングフローに復帰すること | AC-3.3 |
| page=1 命中 + page=2 失効の混在ケースで個別判定が機能すること | AC-3.4 |
| 同条件でソートだけ異なる検索が独立エントリとして扱われること | AC-4.1 |
| ソート往復で元のソートのキャッシュが命中復帰すること | AC-4.2 |
| error-network / error-rate-limited 状態はキャッシュに保存されないこと | AC-5.1 |
| no-results 状態はキャッシュに保存され再検索で即時再現されること | AC-5.2 |
| キャンセルされたリクエストの結果はキャッシュに保存されないこと | AC-5.3 |
| バックグラウンド遷移を理由にキャッシュが破棄されないこと | AC-6.2 |
| 検索画面 pop → 再構築でも同条件キャッシュが再利用されること | AC-6.3 |
| キャッシュキーのタプル一意性（q / sort / order / page のいずれか 1 つ違いで別エントリ） | §5.1 |
| キャッシュエントリ 100 件超過時に最古エントリから破棄されること（FIFO） | §5.4 |
| キャッシュ命中時のユーザー操作から画面反映まで p95 100ms 以内（自動測定可能な範囲で） | §8.1 |

## 11. 将来課題

§2.2 で除外した項目のうち、優先度の高い順に挙げる。

1. stale-while-revalidate（古い結果を即時表示しながら裏で再取得し、差し替える）。
2. in-flight request deduplication（同一クエリの同時実行リクエスト統合）。
3. リポジトリ詳細 / Issue 一覧 / Issue 詳細 / コメントのキャッシュ。
4. ディスク永続キャッシュとプロセス間共有。
5. HTTP 層キャッシュ（`URLCache`）の活用検討。
6. 手動キャッシュクリア UI。
7. キャッシュヒット率テレメトリ。
